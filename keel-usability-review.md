# Keel usability review

A walkthrough of the app's screens from a new user's point of view, focused on one question: is it obvious what to do next, and does the interface ever show someone something they can't actually use? Findings are ordered by how much they'd confuse or block a real user, not by how much code they'd take to fix.

## Findings

**1. The Home page offers a "Reports" link to everyone, but only internal staff can open it.** The landing page's quick-links row (Portfolio Dashboard / All Projects / Reports) shows the same three links regardless of who's logged in. A client-side SUPER_USER can click "Reports," and the page will silently show "No reports yet — generate one now..." — implying they can create one — but the generate button will fail with no visible error, because the underlying API correctly blocks anyone outside your own team. The fix is cheap: hide the Reports quick-link (and the "Weekly report now" / "Steering pack now" buttons) for anyone who isn't internal staff, the same way the sidebar already hides the Reports nav item. Right now the sidebar and the home page disagree with each other about who can see this feature.

**2. The project detail page's Resources tab goes blank for client users with no explanation.** For a client-side SUPER_USER, the "add resource" dropdown is empty (by design — rosters and rates are internal-only), but nothing on screen says why. It reads as a bug, not a boundary. A one-line note ("Staffing is managed by your Keel team") where the empty picker would otherwise sit would turn a confusing dead-end into an understood limitation.

**3. Eleven tabs on the project detail page is a lot to scan, and nothing marks where you are in the lifecycle.** Inception & Ideation, Charter, Tasks, Resources, Status Tracking, Communications, Risks, Milestones, Invoices, Delivery & Pricing, C-Level Report — all sit in one flat, horizontally-scrolling row with equal visual weight. A brand-new user has no way to tell "Charter" and "Delivery & Pricing" are ideation-phase tabs while "Status Tracking" and "Milestones" are execution-phase ones. Grouping the tabs to mirror the same Ideation / Execution / Support split already used in the top-level navigation (even just a visual divider or two-row layout) would make the page feel like it maps to a process instead of a junk drawer of features.

**4. The three top-level entry points (Ideation, Execution, Support) and the Home page's three cards say almost the same thing twice.** Home's cards ("New Ideation," "New Project," "Project Support") duplicate the sidebar's "Project Lifecycle" section, which is fine for a first-time visit but means a returning user re-reads the same three descriptions every time they land on Home before going where they were already going. Consider making Home optional to land on (redirect returning users straight to Dashboard, with Home reachable from the sidebar) or trimming Home to a thinner "what's new / what needs attention" view instead of a full re-explanation of the product.

**5. Nothing on any screen explains the role hierarchy to the person using it.** ADMIN, SUPER_USER, PM, CONTRIBUTOR, VIEWER are all real, meaningfully different roles, but the only place a user encounters that vocabulary is a small role dropdown in Admin. A SUPER_USER logging in for the first time has no in-app explanation of what their role means or what they can/can't do — they'd need to hit a wall (like the Resources tab above) to find out by trial and error. A short one-time tooltip or a line on the Home page ("You're signed in as an Organization Owner — you can see and approve everything in [Org Name]'s projects") would close this gap cheaply.

**6. Empty states are inconsistent in how much guidance they give.** Some are good — the Reports history panel tells you exactly what to do ("generate one now, or wait for the scheduled run"). Others just say "No X yet" with no next step (e.g., several list tabs). Since this is a new product for a new kind of user (boutique consultancies who may not have used a PSA tool before), every empty state is a chance to teach — worth a pass to make sure each one names the specific action that fills it.

**7. The Admin page mixes platform-wide config with per-organization actions in one long scroll.** Organizations, Users & roles, and Automation settings are now three stacked sections on one page — functional, but as organization count grows this will get long fast, and the new deletion-request workflow (confirm/dismiss buttons) sits in the middle of a table a busy admin could scroll past. Given how serious an org deletion is, it might deserve a small banner or badge count ("1 organization has a pending deletion request") near the top of Admin, not just a row-level tag.

## What's already working well

The three-doorway Home page (Ideation / New Project / Support) is a strong, opinionated answer to "what should I do first" — most PM tools dump you on a blank dashboard instead. The sidebar's role-based hiding (Resources/Reports/Admin disappearing for client users) is the right instinct, it just needs to be applied consistently everywhere those same features are linked from (see #1). The export buttons are now consistent and discoverable in the same spot (top-right) on every list screen, which is the kind of small consistency that adds up.

## Suggested priority

Fix #1 first — it's the one place the interface actively promises something that then quietly fails. #2 and #5 are next: both are moments a client user hits a wall with no signpost. #3, #4, #6, and #7 are polish that would matter more as the user base and organization count grow, but aren't blocking anyone today.
